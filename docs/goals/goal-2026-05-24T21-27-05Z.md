# Goal Objective

Generated by pi-goal on 2026-05-24T21:27:05.576Z.

Complete the Pi extension hook/action/filter system so Pi has a lean, efficient, portable harness core, while   
 fork-only/custom behavior attaches through stable extension points instead of scattered core imports.             
                                                                                                                   
   Why:                                                                                                            
   I have a custom Pi fork with many harness customizations, extensions, and tools. I want the core to stay clean, 
 efficient, and easy to keep updated with upstream. The same Pi harness core should run in multiple environments:  
   - MacBook: full custom suite of coding tools, local integrations, agent helpers, UI/extensions.                 
   - lue-kube/OpenClaw pods: same stable Pi harness core with prompt stability/cache optimizations, but without    
 unnecessary local coding tools/extensions.                                                                        
   The goal is portability, environment-specific composition, and minimal upstream merge drift — even though my    
 setup is highly custom.                                                                                           
                                                                                                                   
   Mental model:                                                                                                   
   Use the WordPress-style concept, not a PHP clone:                                                               
   - hook = named lifecycle point                                                                                  
   - action = side-effect callback at a hook, returns nothing                                                      
   - filter = value-transforming callback at a hook, returns transformed value                                     
   - priority default = 10, lower runs earlier, same priority preserves registration order                         
   - callbacks can be removed by hook + id                                                                         
                                                                                                                   
   Design principles:                                                                                              
   - Core owns flow; extensions contribute at named points.                                                        
   - Keep core small and stable.                                                                                   
   - Make custom behavior portable between environments by enabling/disabling extension actions.                   
   - Public API should be agnostic and simple:                                                                     
     - `addAction`                                                                                                 
     - `removeAction`                                                                                              
     - `addFilter`                                                                                                 
     - `removeFilter`                                                                                              
     - `applyFilters`                                                                                              
   - Provide a fluent API:                                                                                         
     ```ts                                                                                                         
     pi.hooks.register("tools:prepare").action("id", callback);                                                    
     pi.hooks.register("systemPrompt:build").filter("id", filter);                                                 
 ```                                                                                                               
                                                                                                                   
 - Also support direct helpers:                                                                                    
   ```ts                                                                                                           
     pi.hooks.addAction("load", "agents", callback);                                                               
     pi.hooks.removeAction("load", "agents");                                                                      
     pi.hooks.addFilter("provider:beforeRequest", "id", filter);                                                   
   ```                                                                                                             
 - Keep pi.tools read-only.                                                                                        
 - Avoid casual runtime tool mutation APIs; tool shaping should happen before session/model prompt freeze to       
   preserve prompt cache affinity.                                                                                 
 - Preserve prompt cache affinity; do not mutate core/cache-affinity.ts.                                           
 - Do not create a huge WordPress-style hook catalog. Add only hooks backed by real existing Pi seams.
