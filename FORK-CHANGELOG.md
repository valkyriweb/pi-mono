Fork feature; logged here per the changelog-ownership split below.
- **Edit queued messages with the up arrow (interactive TUI).** When the agent is busy and you've queued a message, pressing ↑ in an empty editor recalls the most-recently-queued message into the editor for editing (recall priority: compaction-queued → follow-ups → steering). ### Fixed

- **Release publish no longer 422s on the upstream changelog size ("body is too long").** The `Release` workflow used `changesets/action`'s default `createGithubReleases: true`, which builds each package's GitHub Release body by matching a heading whose text equals the version (`## 0.31.0`). The fork's package `CHANGELOG.md` files are upstream-owned Keep-a-Changelog (`## [0.31.0] - 2026-01-02`, `coding-agent` is 400KB+ via `merge=union`), so the match always fails and the action falls back to dumping the **entire file** as the body — exceeding GitHub's 125000-char Release-body cap → `422` → the whole publish job reds (sometimes after publishing to the registry but before every Release is cut). Set `createGithubReleases: false` and added `.github/scripts/create-github-releases.mjs`, a post-publish step that cuts a Release per just-published package (from `steps.changesets.outputs.publishedPackages`) with a small, bounded body pointing at `FORK-CHANGELOG.md` — never the upstream changelog. The upstream-owned changelogs and `merge=union` are left untouched (slimming them is deferred to a separate ownership change with its own ADR). Idempotent (skips packages that already have a Release). Mirrors the same fix in the `my-pi` repo (#139).
- **`biome check` no longer breaks on stray local Pi workspaces.** `.pi/workspaces/` is gitignored globally but not in the repo `.gitignore`, and Biome's `vcs.useIgnoreFile` only reads the repo-local ignore — so a leftover workspace (e.g. `.pi/workspaces/upstream-tti/biome.json` from an upstream-sync session) was discovered as a nested root config and aborted `npm run check` repo-wide. Added `.pi/workspaces/` to the repo `.gitignore` so Biome skips it for everyone (matching the existing `.pi/workflows/` entry).
- **Extension loader resolves upstream package names (`@earendil-works/*`, `@mariozechner/*`).** The fork renamed its packages to `@valkyriweb/*`, but third-party extensions in the ecosystem still import the upstream scopes — `@earendil-works/*` (current upstream) and `@mariozechner/*` (legacy). Type-only imports erase at runtime, but extensions with **value** imports (e.g. pi-autoresearch pulling `truncateTail`/`formatSize`/`StringEnum`/`Text` as runtime values) failed to load with `Cannot find module "@mariozechner/pi-coding-agent"`. Added compatibility entries for both upstream scopes mapping onto the same fork modules, in **both** resolution paths: `getAliases()` (jiti, Node/dev mode) and `VIRTUAL_MODULES` (the compiled Bun binary that live `pi` actually runs — line 682 selects `virtualModules` over `alias` when `isBunBinary`). Covers `pi-coding-agent`, `pi-agent-core`, `pi-tui`, `pi-ai`, and `pi-ai/oauth`. Generic platform compatibility primitive (extension module resolution) that can't be expressed through hooks — restores the third-party extension ecosystem on the fork without each extension needing a `@valkyriweb/*` rewrite.
- **`setActiveToolsByName` cache-burst no-op guard (cache law #4).** Before rewriting `agent.state.tools` and rebuilding the system prompt, the method now early-returns when the freshly built `tools[]` is element-for-element **reference-identical** to the current state. An unconditional refresh on every `tools_changed`/`_refreshToolRegistry` event bursts the within-session `tools[]` (and system-prompt) cache prefix even when nothing actually changed. Reference-equality (not name-set equality) is the correct test: pi rebuilds the registry with fresh object refs whenever a definition changes (a tool gains `deferLoading`, an allowlist empties), so a genuine change yields different refs and the rebuild proceeds — only a true no-op early-returns (mirrors pi-ai's `convertedToolCache` WeakMap keyed by tool reference). Guard is skipped until the base prompt has been built once (`_baseSystemPromptOptions` is set only by `_rebuildSystemPrompt`), so the cold init `empty→empty` rebuild still runs. Same change adds the post-registration **deferral-override seam** (`setDeferredToolOverrides`/`_applyDeferredOverrides`): extensions can force already-registered tools to `deferLoading:true` so they ride `tools[]` as `defer_loading` stubs across every registry rebuild, with the same unchanged-set no-op guard (`sameStringSet`) so a redundant call doesn't burst the prefix. Also threads the live agent state into `createPromptCacheAffinityKey` on every model switch so the affinity key tracks the actual model rather than a stale cwd-only key.
- **Changelog guard: package-level test-only changes no longer wrongly require a CHANGELOG.** `requiredChangelog()` in `scripts/check-changelog-updated.mjs` returned the package `CHANGELOG.md` for any `packages/<pkg>/` path before consulting `isIgnored()` (which already covers test-only paths via `isTestOnly()`), so a change touching only `packages/<pkg>/test/**` failed the guard, contradicting the script's own "tests-only ... are ignored" message and the AGENTS changelog rules. Now `isIgnored(path)` is checked first (returns no requirement), then package/fork resolution. Verified with crafted diffs: a package test-only change passes, a package src change without its CHANGELOG still fails.
- **Workflow shellcheck hygiene: quoted `$(which fdfind)` and grouped `$GITHUB_OUTPUT` redirects so `actionlint` (incl. shellcheck) is clean.** SC2046 (word-splitting on unquoted command substitution) in `ci.yml`, `build-binaries.yml`, and `release.yml`; SC2129 (consecutive redirects) in the `upstream-sync.yml` conflict branch. Behavior-preserving; lets the new `workflow-sanity.yml` gate pass on day one.
- **`fork-safety-check` CI: reordered `npm run build` so the four workspace packages build before `test:build-gate` runs.** The root `build` script ran `test:build-gate` first, but the gate's `test:e2e` step (`packages/coding-agent` `test/suite`) imports `@valkyriweb/pi-tui/dist/index.js` via `modes/interactive/theme/theme.ts`. Because `packages/tui` had not been built yet, the gate crashed with `Cannot find package '.../@valkyriweb/pi-tui/dist/index.js'`, failing the `Build workspace declarations` step on every PR. New order: `tui → ai → agent → coding-agent`, then `test:build-gate` last, so the gate runs against freshly built dist (its intent — catching prompt-cache/system-prompt regressions before a build ships — is unchanged). Also guarded the gate's `test:my-pi-extensions` step (`npm --prefix ../my-pi run test:extension-gate:ci`) to skip when the `my-pi` sibling repo is absent: on CI the sibling is not checked out, so the step now fails with `ENOENT .../my-pi/package.json` once the earlier crash is gone. The guard runs the extension gate locally (where `../my-pi` exists) and skips with a notice on CI. Finally, gave the `fork-safety-check` Checkout `fetch-depth: 0` so the job's `npm run check` step (`check:changelog` runs `git diff origin/main...HEAD`) can resolve `origin/main` on CI — the previous shallow checkout made that diff fail with `unknown revision origin/main` once the build step stopped crashing first (mirrors the standalone `changelog.yml` workflow, which already uses `fetch-depth: 0`).

- **Bash foreground-timeout moved behind an additive `onBashTimeout` seam; core default is now upstream-neutral (kill).** The exec loop no longer hardcodes a detach policy; on timeout it consults a pluggable resolver given a `BashTimeout` context (`command`, `cwd`, `timeoutMs`, `background()`, `kill()`) and acts on the `BashTimeoutOutcome` (`{ backgroundedJobId }` or `{ failed }`). The **core default kills and reports `Command timed out after Ns`** (closest to upstream); the opinionated detach-on-timeout policy lives in the my-pi `native-tool-aliases` extension, which calls the exported `onBashTimeout((t) => ({ backgroundedJobId: t.background().id }))` so Luke's `Bash` tool auto-backgrounds (Claude Code parity) while core stays neutral. Consumers override via `onBashTimeout(resolver)` (returns a restore fn) to detach, kill, log, or gate — without reshaping the upstream exec path (one call deep). Honours the additive-hooks principle: `core/tools/bash.ts` keeps its shape, the policy lives in the extension. Exported `onBashTimeout` + `BashTimeout`/`BashTimeoutOutcome` types from the package barrel. Seamed e2e coverage in `test/bash-timeout-background.test.ts`: default kill+timeout report, override-to-detach + bg read, and override-to-fail (seam sees real command/timeout, restore returns to default).
- **Fork-mode sub-agent prefix: switched from strip to placeholder-substitute (CC parity).** `getFilteredForkMessages` in `packages/coding-agent/src/core/agents/context.ts` used to drop parent `agent`/`subagent` `tool_use` blocks (and any associated `tool_result`s) from the fork child's message history. That broke prompt-cache continuity: the moment a stripped tool_use appeared in parent's actual cached prefix, every subsequent block in the child's request diverged byte-for-byte from the parent's cache key, busting cache reuse for the entire leading section. New behavior mirrors Claude Code 2.1.x `buildForkedMessages` (`src/tools/AgentTool/forkSubagent.ts`): keep parent's `tool_use` blocks in place, and for any unresolved `tool_use` (denied or simply incomplete at fork time) synthesize a fixed-bytes placeholder `tool_result` with text `"Sibling agent task in progress."`. New helper `substitutePlaceholdersForUnresolvedToolCalls` is exported alongside `getFilteredForkMessages`. The placeholder text is identical across every fork child spawned from the same parent state, so **sibling forks share cache with each other**, not just with the parent — the critical property for wide parallel fan-out via `tasks[]`. Cache parity contract: any two `getFilteredForkMessages(parentSession)` calls return byte-identical arrays through every leading block, with the per-child user directive (built by `buildChildTaskPrompt`) as the only divergence point. Removed the now-dead `filterDeniedToolArtifacts` helper; left a comment marker so future maintainers see the historical rationale. Regression tests in `packages/coding-agent/test/agent-context-inheritance.test.ts`: "fork filtering keeps parent agent/subagent tool_uses for cache parity", "fork filtering substitutes placeholders for unresolved tool_uses, drops orphan results", and "sibling forks produce byte-identical message arrays" (JSON.stringify equality on two consecutive `getFilteredForkMessages` calls). Side note: orphan `tool_result`s (results whose `tool_use` doesn't exist in the message stream) are still dropped — those would confuse the provider API and don't appear in parent's cached prefix anyway.

- **`maxTurns` hard turn cap for the agent loop, surfaced through `ctx.forkAgent({ maxTurns })`** (mirrors Claude Code's `query({ maxTurns })`). A new first-class field on `AgentLoopConfig` (`packages/agent/src/types.ts`) and the `Agent` engine (`AgentOptions.maxTurns` + public `Agent.maxTurns`, threaded into `createLoopConfig`). `runLoop` (`packages/agent/src/agent-loop.ts`) counts completed assistant turns and, when the count reaches `maxTurns`, emits `agent_end` and returns **before** starting another LLM call — even if the model still wants to call tools — giving a deterministic safety bound for runaway agentic forks (e.g. background memory extractors). Plumbed into coding-agent via `AgentTaskConfig.maxTurns` (`core/agents/types.ts`), set on the child engine in `runChild` (`core/agents/executor.ts`) before driving, exposed on `ForkAgentOptions.maxTurns` (`core/extensions/types.ts`), and mapped through `engine.fork` (`core/agents/engine.ts`). - **`ToolDefinition.replacesBuiltins` — extensions can drop core base builtins they supersede.** An extension-registered override declares the lowercase base builtins it fully replaces (e.g. my-pi `native-tool-overrides`' `Read`/`Edit`/… declare `["read"]`/`["edit"]`/… and `TaskOutput`/`TaskStop` declare `["bash_output"]`/`["bash_kill"]`). `AgentSession._refreshToolRegistry` collects the declarations from the live extension tool set and drops the named builtins from both `_toolDefinitions` (so `getAllTools()` omits them) and `_toolRegistry` (so they can't be activated), so a consumer that force-activates the whole registry (e.g. rusty-wa) no longer surfaces duplicate `Read`/`read` schemas. No declaration ⇒ base tools stay, so vanilla/upstream sessions are unaffected. Resolved once at registry build (no per-turn state) so the `tools[]` prefix stays cache-stable. Optional field on `ToolDefinition` (`core/extensions/types.ts`) + filter in `core/agent-session.ts`. Regression: `test/suite/regressions/replaces-builtins.test.ts` (drops when declared, keeps when not); fork `test:build-gate` green.
- **Auto-background-on-timeout for the built-in `bash` tool** (mirrors Claude Code's `tengu_bash_command_timeout_backgrounded`). Instead of killing a timed-out foreground command, the exec layer now **detaches the still-running process** and adopts it into the background-job registry (`adoptBashBackground` in `core/tools/bash.ts`): a race between real exit and the timeout resolves to `{ exitCode: null, backgroundedJobId }`, the foreground `onData` listeners are detached, future stdout/stderr is redirected to `~/.pi/agent/bash-bg/<bgId>.log`, and the adopted child's pid stays tracked (the bg job owns it). The result reports the `bgId`/`pid` and that the process was **not** killed. Read live output with `bash_output(bgId)`, stop with `bash_kill(bgId)`. The `timeout` param description documents the behavior. (Custom `BashOperations` that still throw the legacy `timeout:` sentinel fall back to a plain timeout error.)

### Fixed

- Made the built-in TUI footer context count use an extension-owned cached snapshot that includes the effective system prompt, active provider tool schemas, and transcript/provider usage. Provider tool schemas are top-level request `tools`, not system-prompt text, but still count against the context window; deferred tool index text counts only as prompt text until the full tool schema is active.
- Resolved Anthropic `stop_reason: "pause_turn"` **in-stream** inside `streamAnthropic` instead of mapping it to a terminal `"stop"` (`packages/ai/src/providers/anthropic.ts`). When a long-running server-side tool (`web_search` / `web_fetch` / `code_execution`) needs another turn to finish, Anthropic returns a partial assistant message and the client must echo it back unmodified to resume. The previous mapping let the partial turn reach the agent loop, which persisted it to the session JSONL. On the next user prompt or auto-compaction the partial turn replayed as the latest assistant message and any byte-level drift in a thinking signature triggered a hard 400 (`messages.N.content.M: \"thinking\" or \"redacted_thinking\" blocks in the latest assistant message cannot be modified`) that then poisoned every subsequent request, including compaction. Fix: capture the raw stop reason in `message_delta`, and while it's `pause_turn` (capped at 16 resumes) append the partial assistant message to `context.messages` and re-issue `client.messages.create` with the same `options.onPayload` hook and `requestOptions`. Signed thinking blocks round-trip byte-for-byte through the existing `transformMessages` + assistant-block builder path (`packages/ai/src/providers/anthropic.ts:~1163`), so the resumed request passes Anthropic's signature validation. The `mapStopReason("pause_turn")` arm stays as a defensive fallback (`"stop"`) for any caller wiring `mapStopReason` directly. Regression coverage: `packages/ai/test/anthropic-pause-turn-resume.test.ts` (single-resume happy path with thinking-signature preservation in the continuation payload; multi-resume loop walking three `pause_turn` responses before terminating; resume-limit cap surfacing an `"error"` stop reason with a clear message). (#thinking-roundtrip)
- Documented the **OpenClaw + pi-harness model-namespace ambiguity trap** in [`docs/openclaw-pi-harness-debug.md`](docs/openclaw-pi-harness-debug.md).
